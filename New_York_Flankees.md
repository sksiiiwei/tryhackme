### Взятие шелла
Начинаем стандартно - со скана открытых портов (сначала прошлась masscan'ом по tcp/udp портам, потом уже воспользовалась nmap'ом). По UDP у нас ничего интересного,
по TCP два порта. Один из них - сайт на нестандартном порту 

<img width="694" height="247" alt="image" src="https://github.com/user-attachments/assets/e5ebed20-9040-4389-be84-076d4aece372" />

Сразу запустила фаззер директорий (через ffuf: ```ffuf -u http://ip:8080/FUZZ -w rockyou.txt -e .html -fs 4332```, смотрела с расширением,
тк когда зашла на сайт, увидела, что некоторые страницы имеют расширение html). Также пофаззила сабдомены - пусто. Из интересного он выдал:

<img width="792" height="83" alt="image" src="https://github.com/user-attachments/assets/0a17b876-9ca8-4e8e-8efa-2e7c0ebac1b9" />

Получается, что у нас, вероятнее всего, есть какая-то командная строка по адресу /exec.html. Для доступа, предположительно, надо залогиниться на /login.html.
Сразу решила прогнать форму через sqlmap (req - сохраненный файл запроса из бурпа, ignore code добавила из-за редиректа). Результатов не дало
```
sqlmap -r req  \
--batch \       
--random-agent --ignore-code 401 \
--threads 5 \
--level 3 --risk 2 \ # попробовала и с этими флагами, после того как sqlmap без них поля определил как неуязвимые
```
Поля к sqli неуязвимы - окей, идем дальше. Попробовала посмотреть код страницы и увидела вот такой интересный скрипт:

Сразу попробовала изменить куки на что-то вроде loggedin=true, кнопка Admin Login действительно изменилась на Debug и теперь ведет на /exec.html, но страница все также
отвечает кодом 401

<img width="1916" height="379" alt="image" src="https://github.com/user-attachments/assets/331327c4-d024-43e9-a73e-e5a0b3147b56" />

Очевидно, что на сервере проверяются куки и они должны выглядеть определенным образом. Т.к. у нас есть страница /debug.html решила посмотреть там.
Там код страницы тоже порадовал, ведь мы буквально видим инструкцию по получению токена

<img width="716" height="257" alt="image" src="https://github.com/user-attachments/assets/45b0a34d-b1f5-4e08-a9c1-03b6101a0912" />

Переходим - токен мы, к сожалению, не получаем. Но как вариант - нам нужно расшифровать предоставленную строку. Алгоритм нам предоставлен - AES/CDC/PCKS.
С блочным шифрованием нам поможет утилита padbuster

``` padbuster "http://10.113.187.37:8080/api/debug/value" "value" 16 -encoding 2 ```

После расшифровки из нее мы получаем креды для входа - и вот мы stephan1197 c доступом к командной строке
