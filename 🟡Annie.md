# принцип работы эксплойта:
стадия 1: использование протокола обнаружения (discovery)
      скрипт формирует пакет, который выглядит как стандартное «приветствие» от другого клиента энидеск. пакет содержит поля: ID клиента, ос, имя пользователя и хостнейм(имя компа).
стадия 2: Уязвимость Format String
      когда AnyDesk получает этот пакет, он должен как-то обработать присланные данные (например, записать в лог: «обнаружен пользователь x на компьютере y»). и здесь в коде была допущена критическая ошибка. примерно такая (упрощенно):
      ```
      // вместо безопасного:
      printf("User detected: %s", received_hostname); 
      // написано:
      printf(received_hostname);
      ```
      если received_hostname — это просто текст, всё ок. но если в нем есть символы %, функция printf начинает интерпретировать их как команды.
стадия 3: манипуляция стеком
      скрипт отправляет в поле hostname строку: %1$*1$x%18x%165$ln. символ % заставляет программу не печатать текст, а смотреть в свой стек (временную память, где хранятся переменные и адреса возврата функций). число 165 в скрипте — это оффсет, порядковый номер ячейки в стеке, где лежит адрес, по которому программа собирается что-то записать.
стадия 4: запись в память через %n
      спецификатор %n (или %ln для длинных чисел) записывает данные. он берет количество символов, которые были "напечатаны" функцией printf до этого момента, и записывает это число по адресу, указанному в стеке. скрипт подбирает количество символов (через %18x и прочие конструкции) так, чтобы это число в сумме составило нужный нам адрес в памяти (адрес шелл-кода).
      итог этой стадии: AnyDesk взяла адрес в своей памяти и перезаписала его нужным нам значением. обычно перезаписывается адрес возврата (Return Address) функции или запись в Global Offset Table (GOT).
стадия 5: перехват управления
      когда функция обработки пакета в AnyDesk заканчивает работу, она должна вернуться к основному коду. для этого она смотрит в стек, чтобы понять, че выполнять дальше. в нормальной ситуации там адрес следующей легитимной инструкции, но мы с помощью %n только что подменили этот адрес на адрес нашего шелл-кода, который тоже лежит в отправленном нами пакете. процессор послушно прыгает по адресу и начинает выполнять нужные байты (\x48\x31\xc9...).
Стадия 6: Выполнение шелл-кода
      шелл-код, сгенерированный через msfvenom, делает следующее: использует системные вызовы Linux (syscall), чтобы создать новый сетевой сокет, подключается к нашему айпи по нужному порту, дублирует стандартные потоки ввода/вывода (клавиатура и экран) в это сетевое соединение, запускает /bin/sh (командную оболочку).

1) ```root       439  0.0  6.1 511952 28880 ?        Sl   11:44   0:00  _ /usr/bin/Xvfb :10 -ac -screen 0 1024x768x24```
!!!флаг -ac (Disable Access Control). в нормальном состоянии, чтобы подключиться к графическому экрану, нужен секретный файл (.Xauthority).с флагом -ac любой пользователь в системе может подключиться к этому виртуальному экрану без пароля и ключей. еще и запущено от рута (AnyDesk для работы на серверах без монитора часто сам поднимает Xvfb, чтобы ему было что транслировать удаленному пользователю)
а это значит, что можно сделать скриншот того, что происходит на дисплее :10 (```xwd -display :10 -root -out screen.xwd```), перехватить нажатий клавиш через xinput, инъектировать нажатия (```export DISPLAY=:10; xdotool key Return```). вообще, кейс интересный, хотя на тренировочной тачке мало чем полезен;(
идем дальше:
2) ```net.ipv4.conf.all.rp_filter = 1```
rp_filter (Reverse Path Filter) - механизм защиты от IP Spoofing (подмены айпи адреса). когда сервер получает пакет, он делает проверку: смотрит на адрес отправителя (Source IP), заглядывает в таблицу маршрутизации и проверяет: «если бы я сам отправлял ответ на этот айпи, через какой интерфейс (карту) я бы его погнал?», если ответ должен уйти через тот же самый интерфейс, через который пришел пакет - всё ок, но если пакет пришел через интерфейс eth0, а по таблице маршрутизации ответ должен уйти через tun0 — значит, пакет подозрительный (подделанный), и сервер его удаляет.
0 - ильтр выключен.
1 (наш случай!) - Strict Mode (Строгий режим). самый безопасный вариант.
2 - Loose Mode (мягкий режим). проверяет только, есть ли вообще путь к этому IP.
линпис это подсветил тк это:
1. хардеринг (усиление защиты)
значит, что классические атаки типа Man-in-the-Middle или попытки отправить пакет с фейкового айпи внутри этой сети, скорее всего, провалятся. сервер просто будет дропать пакеты.
2. препятствие для туннелирования (пивотинга)
использовать эту взломанную машину как «шлюз», чтобы через неё атаковать другие машины во внутренней сети, будет сложно. эта настройка будет мешать
